# Id is authoritative, name is a hint

Humans think in names. Computers should store ids. The resolution between the two is a rule, not a negotiation:

- **Ids never collide.** 64 hex characters of BLAKE3 → no meaningful chance of accidental duplication.
- **Names can collide.** If two kinos both claim the name `greeting`, that is a human-level ambiguity. The system must surface it, not guess.

So the rule is:

- On *store*, the author may write names in kinograph entries. The store command resolves each name against the current ledger and rewrites the entry with the resolved id. The on-disk blob is thereafter authoritative by id.
- On *resolve*, lookup by name is allowed but fails loudly on ambiguity: "name `greeting` is ambiguous; matches 2 identities: abc…, def…"
- Names preserved alongside ids on kinograph entries are *hints* — they round-trip but carry no semantics. Renaming the referenced kino later never breaks the kinograph.

This is the same principle as git refs: branches are movable labels, but every tool internally thinks in object ids. Names give you discoverability and a memorable handle; ids give you correctness.
