# Kinographs compose by reference, not by copy

A *kinograph* is itself a kino — `kind: kinograph` — whose content is a styx document listing entries:

```styx
entries (
  {id b3aaa…}
  {id b3bbb…, name content-addressing, note "atomic concept"}
  {id b3ccc…, pin b3xxx…}
)
```

Each entry is a pointer. The referenced kino is not copied into the kinograph's blob; it is *inlined at render time* by following the id (or a pinned version hash) and fetching the current content.

This has several consequences:

- **The same kino can appear in many kinographs.** There is no ownership conflict — kinographs are composition, not containment.
- **Renaming a kino never breaks a kinograph.** Ids are authoritative; names are hints.
- **A kinograph's blob stays small and stable.** Editing the content of an entry does not invalidate the kinograph.
- **Pinning is opt-in.** Leave `pin` off and the entry tracks the current head; set it to freeze the entry to a specific version.

Kinographs can reference kinographs. The rendered result is a tree flattened depth-first — which lets a docs site, a mosaic, or a hierarchical table of contents all be described in the same primitive.
