# The event envelope

Every ledger event carries a fixed envelope:

```json
{
  "kind": "markdown",
  "id":   "<identity-hash>",
  "hash": "<content-hash>",
  "parents": ["<hash>", ...],
  "ts":   "2026-04-18T09:20:00Z",
  "author": "yj",
  "provenance": "where this came from",
  "metadata": { "name": "greeting", ... }
}
```

Each field earns its place:

- **`kind`** — what this kino *is*. Reserved values (`markdown`, `text`, `binary`, `kinograph`) or a namespaced extension (`prefix::name`). Enables type-aware tools without forcing every kino through the same lens.
- **`id`** — the *identity*. Stable across all versions of a kino. Born as the content hash of the first version.
- **`hash`** — the *version*. The content hash of this particular version's blob.
- **`parents`** — the content hashes that influenced this version. Shapes the version DAG: linear (one parent), fork (zero, same identity), merge (many).
- **`ts`** — timestamp of the write.
- **`author`** — who wrote this. Derived from git if not given; error if unresolvable.
- **`provenance`** — a free-form string saying *why* this content exists (a URL, a prior kino, a conversation ID). Never empty.
- **`metadata`** — everything else, with namespace rules.

The envelope is canonical: JSON with sorted keys, stable field order. Two systems that see the same logical event will always compute the same event hash.
